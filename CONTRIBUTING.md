# Contributing to behaverse/schemas

This repo publishes metadata **schemas** for the Behaverse data ecosystem at
`https://behaverse.org/schemas/<name>`. Each schema lives in its own directory
(`bcsv/`, `catalog/`, `dataset/`, `studyflow/`, `trial/`, `event/`) and ships a
`schema.json`, a `context.jsonld` (where applicable), examples, a `README.md`, a
`CHANGELOG.md`, and per-version snapshots under `versions/`.

## Source of truth — know which file to edit

Schemas differ in how their artifacts are produced. **Edit the source, never the
generated file.** A CI guard (below) fails the build if a generated file drifts from
its source.

| Schema | Source of truth | Generated by | Generated artifacts |
|---|---|---|---|
| **catalog**, **dataset** | `field-definitions.yaml` | `<schema>/scripts/generate_schema_files.py` (a thin shim over the shared `scripts/schema_gen.py`) | `schema.json`, `context.jsonld` |
| **trial** | `field-definitions.yaml` | `trial/scripts/generate.py` | `field-definitions.json` (BDM render artifact), `schema.json` |
| **event** | `field-definitions.yaml` **and** hand-maintained `schema.json` / `context.jsonld` | `event/scripts/generate.py` | `field-definitions.json` (only) |
| **bcsv** | hand-maintained `schema.json` + `context.jsonld` | — | — |
| **studyflow** | hand-maintained `schema.linkml.yaml` (LinkML) | LinkML toolchain | — |

> **event caveat:** its `schema.json` and `context.jsonld` are **hand-maintained**, not
> generated from the YAML — keep them in sync by hand. trial's `schema.json` validates
> *types + required fields only* (the YAML's `type`/`range` are coarse); it does not
> enforce enum values, numeric ranges, or foreign keys.

## Authoring workflow

1. **Edit the source of truth** for the schema (the YAML for catalog/dataset/trial/event;
   `schema.json` directly for bcsv).
2. **Regenerate** (for the generated schemas):
   ```bash
   # catalog / dataset
   python catalog/scripts/generate_schema_files.py        # or dataset/...
   # trial / event  (uv reads the inline script deps; plain python works too)
   uv run trial/scripts/generate.py                       # or event/...
   ```
3. **Update `examples/`** if the change affects them, and add a `CHANGELOG.md` entry.
4. **Version** (for a released change): bump `version` + `$id`, snapshot the outgoing
   version to `versions/vYY.MMDD/`, and add the CHANGELOG entry. CalVer `vYY.MMDD`. See
   [`VERSIONING.md`](VERSIONING.md).
5. **Validate locally** (this is exactly what CI runs):
   ```bash
   pip install pyyaml jsonschema
   python scripts/validate_schemas.py                     # well-formedness + examples
   python catalog/scripts/generate_schema_files.py --check  # drift guard (no diff)
   python dataset/scripts/generate_schema_files.py --check
   python trial/scripts/generate.py --check
   python event/scripts/generate.py --check
   ```

## CI & deployment

`/.github/workflows/validate-schemas.yml` runs on every PR and push to `main`:

1. **Drift guard** — runs each generator in `--check` mode; fails if any generated file
   is out of sync with its `field-definitions.yaml`.
2. **Validation** — `scripts/validate_schemas.py` checks every `schema.json` is
   well-formed (Draft-07 or 2020-12, per its `$schema`) and that its `examples/*` validate.
3. **Deploy** — on a **push to `main`** (after 1–2 pass), it triggers the docs deploy.
   **So merging to `main` publishes the site automatically** — you no longer need to
   trigger a deploy by hand.

The Docusaurus site itself lives on the **`gh-pages`** branch (its `docs/`,
`scripts/generate_docs.py`, and `deploy-docs.yml`). The deploy fetches the schema content
from `main` at build time, so schema edits only need to land on `main`. To deploy manually
(e.g. after editing the site on `gh-pages`):

```bash
gh workflow run deploy-docs.yml --ref gh-pages
```

Verify a deploy against `https://behaverse.github.io/schemas/...` first — `behaverse.org`
is Cloudflare/Varnish-cached (~10 min) and lags.

## Conventions

- **CalVer** `vYY.MMDD`; snapshots under `versions/` are immutable.
- Don't reference sibling consumer packages (`bcsv-py`, `bcsv-r`) in user-facing docs
  until they ship.
- Keep schema constraints reasonably flat (deeply-nested `allOf` can confuse some
  consumer validators).
